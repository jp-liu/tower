  <action>
Create `src/mcp/tools/terminal-tools.ts` following the exact same pattern as `task-tools.ts` (export a `terminalTools` object with `description`, `schema`, `handler` per tool).

All three tools call the Phase 33 HTTP bridge via `fetch("http://localhost:3000/api/internal/terminal/${taskId}/...")`. The MCP stdio process runs in a separate process from Next.js, so it CANNOT import session-store directly (separate globalThis). HTTP bridge is the only correct IPC channel.

Define a module-level constant: `const BRIDGE_BASE = "http://localhost:3000/api/internal/terminal";`

Helper function at module level:
```typescript
async function bridgeFetch(path: string, init?: RequestInit): Promise<Response> {
  return fetch(`${BRIDGE_BASE}${path}`, init);
}
```

**Tool 1: `get_task_terminal_output`** (TERM-03)
- Description: "Get recent terminal output lines from a running task's PTY session. Returns the last N lines (default 50, max 500)."
- Schema: `{ taskId: z.string(), lines: z.number().int().min(1).max(500).optional() }`
- Handler:
  1. Call `bridgeFetch(`/${args.taskId}/buffer?lines=${args.lines ?? 50}`)`
  2. If response.status === 404, return `{ error: "No active terminal session for this task", taskId: args.taskId }`
  3. If response.ok, parse JSON and return `{ taskId, lines, total, killed }` (pass through the bridge response)
  4. Otherwise return `{ error: "Bridge request failed", status: response.status }`

**Tool 2: `send_task_terminal_input`** (TERM-04)
- Description: "Send text input to a running task's PTY terminal. The text is forwarded directly to the Claude CLI process. Include \\n for Enter key."
- Schema: `{ taskId: z.string(), text: z.string().min(1).max(10000) }`
- Handler:
  1. Call `bridgeFetch(`/${args.taskId}/input`, { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify({ text: args.text }) })`
  2. If response.status === 404, return `{ error: "No active terminal session for this task", taskId: args.taskId }`
  3. If response.status === 410, return `{ error: "Terminal session has exited", taskId: args.taskId }`
  4. If response.ok, return `{ ok: true, taskId: args.taskId }`
  5. Otherwise return `{ error: "Bridge request failed", status: response.status }`

**Tool 3: `get_task_execution_status`** (TERM-05)
- Description: "Get the execution status of a task including whether its terminal is running, idle, or exited. Also returns a snippet of recent output."
- Schema: `{ taskId: z.string() }`
- Handler:
  1. Import `db` from `"../db"` (same as other tool files)
  2. Query latest execution: `db.taskExecution.findFirst({ where: { taskId: args.taskId }, orderBy: { createdAt: "desc" } })`
  3. If no execution found, return `{ error: "No execution found for this task", taskId: args.taskId }`
  4. Call `bridgeFetch(`/${args.taskId}/buffer?lines=10`)` to get recent output
  5. Determine `terminalStatus`:
     - If bridge returns 404 (no active session): check execution.status — if COMPLETED/FAILED use "exited", else "not_running"
     - If bridge returns 200: parse response, if `killed` is true then "exited", else "running"
  6. Build snippet: if bridge returned lines, join last 5 lines with `\n`; otherwise `null`
  7. Return:
     ```typescript
     {
       taskId: args.taskId,
       executionId: execution.id,
       executionStatus: execution.status,  // DB status: PENDING/RUNNING/PAUSED/COMPLETED/FAILED
       terminalStatus,                      // live status: running/exited/not_running
       startedAt: execution.startedAt,
       endedAt: execution.endedAt,
       outputSnippet: snippet,
     }
     ```

Do NOT import anything from `@/lib/pty/` — all PTY access goes through HTTP bridge.
Use `import { z } from "zod"` and `import { db } from "../db"` only.
