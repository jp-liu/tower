## What Was Built

**MissionCard (`src/components/missions/mission-card.tsx`):**
- `useSortable({ id: execution.executionId })` with CSS.Transform.toString for smooth drag transforms
- Header: drag handle button (GripVertical, cursor-grab), breadcrumb `WS > Project > Task` with `projectAlias || projectName` fallback, status badge, running time, open full view button (router.push), stop button
- Status badge distinction per D-11: RUNNING badge (primary/blue) when active; "Stopped" badge (muted) when `removeReason === "stopped"`; "Completed" badge (muted) when `removeReason === "completed"`
- Live elapsed timer: `useState` + `useEffect(setInterval(1000))`, cleared on unmount and when `isRemoving=true`
- TaskTerminal loaded via `next/dynamic({ ssr: false })` — xterm.js requires browser environment
- `worktreePath ?? projectLocalPath` passed to TaskTerminal — avoids "no worktree" state (Research pitfall #1)

**TaskPickerDialog (`src/components/missions/task-picker-dialog.tsx`):**
- Three cascading selects: Workspace → Project → Task, all using manual `<span>` (never `<SelectValue>`) per ui.md
- Workspace loads on dialog `open`; selections reset on each open
- Project list derived from selected workspace (no extra fetch)
- Task list fetched via `getProjectTasks()` when project changes, filtered to `TODO | IN_PROGRESS`
- Launch: `startPtyExecution(selectedTaskId, "")` → `onLaunched()` → `onOpenChange(false)` with `toast.error` on failure

**MissionsClient (`src/app/missions/missions-client.tsx`):**
- `removingIds`: `Map<string, "stopped" | "completed">` tracks fading cards AND their removal reason
- `removingIdsRef`: `useRef` mirror kept in sync via `useEffect([removingIds])` — enables stale-closure-free polling
- `startFadeOut(executionId, reason)`: guarded by ref check, sets Map entry, 3s setTimeout removes card from DOM
- Polling `useEffect` (4000ms): detects gone cards → `startFadeOut(..., "completed")`; appends newly appeared cards; depends only on `[startFadeOut]` (stable), so interval is NOT torn down on `removingIds` change
- `handleStop(taskId)`: optimistic `startFadeOut(..., "stopped")` before `stopPtyExecution` API call
- `handleSessionEnd`: terminal exit → `startFadeOut(..., "completed")`
- `handleLaunched`: immediate poll to append new execution without waiting 4s
- Unmount cleanup: `fadeTimers.current.forEach(clearTimeout)` per process-lifecycle.md
- Empty state: only shown when `cards.length === 0 && removingIds.size === 0` — no flash during last card fade

## Decisions Made

- `removingIds` is `Map<string, "stopped"|"completed">` not `Set<string>` — the Map value carries the removal reason (D-11 requires distinguishing "Stopped" vs "Completed" badge text per UI-SPEC)
- `removingIdsRef` useRef mirror pattern chosen over including `removingIds` in polling deps — adding `removingIds` to `useEffect([startFadeOut, removingIds])` would tear down and recreate the `setInterval` on every card fade, causing 4s gap jitter; ref avoids this
- `startFadeOut` has empty `useCallback([])` dep array — reads `removingIds` exclusively through ref, making the function reference stable and safe as polling `useEffect` dependency
- `worktreePath ?? projectLocalPath` for TaskTerminal — Research pitfall #1: when execution starts before worktree is created, `worktreePath` may be null; falling back to `projectLocalPath` provides a valid cwd
