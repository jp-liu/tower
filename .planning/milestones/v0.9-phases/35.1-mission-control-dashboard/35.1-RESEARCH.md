## Summary

Phase 35.1 builds a "Mission Control" dashboard at `/missions` that aggregates all RUNNING task executions across every workspace into a single paged grid. The page renders an xterm.js terminal inside each card, supports a fixed set of grid preset layouts, and enables the user to launch new executions or stop existing ones — without leaving the page.

All core primitives are already present in the project: xterm.js + FitAddon via `TaskTerminal`, WebSocket PTY server on port 3001 (via ws-server.ts), dnd-kit for drag-and-drop, Shadcn Dialog and Select for the launcher, and `startPtyExecution`/`stopPtyExecution` for lifecycle control. This phase is primarily wiring and new UI — no new libraries are needed.

The key complexity lies in three areas: (1) the `getActiveExecutionsAcrossWorkspaces` server action that crosses workspace boundaries, (2) correctly reusing `TaskTerminal` which requires `worktreePath` but should be adapted (or wrapped) for mission-card use since execution may have no worktree, and (3) the poll-driven auto-remove lifecycle that must not flicker when executions complete within the 3-second badge window.

**Primary recommendation:** Follow the canonical file patterns exactly. Build Plan 01 (server action + route skeleton + sidebar) first — the remaining two plans depend on it. Reuse `TaskTerminal` unchanged; wrap each card in `next/dynamic({ ssr: false })`.

### Pattern 1: Server Action — getActiveExecutionsAcrossWorkspaces

**What:** Prisma query that finds all TaskExecutions with `status: "RUNNING"`, joining Task → Project → Workspace.

**When to use:** Called on initial page load (server component) and polled every ~4 seconds by the client component.

**Return shape:**
```typescript
// Source: derived from existing agent-actions.ts Prisma patterns
interface ActiveExecution {
  executionId: string;
  taskId: string;
  taskTitle: string;
  projectId: string;
  projectName: string;
  projectAlias: string | null;
  workspaceId: string;
  workspaceName: string;
  worktreePath: string | null;
  startedAt: Date | null;
}

// Server action implementation pattern
export async function getActiveExecutionsAcrossWorkspaces(): Promise<ActiveExecution[]> {
  const executions = await db.taskExecution.findMany({
    where: { status: "RUNNING" },
    orderBy: { startedAt: "asc" },
    include: {
      task: {
        include: {
          project: {
            include: { workspace: true },
          },
        },
      },
    },
  });
  return executions.map((e) => ({
    executionId: e.id,
    taskId: e.taskId,
    taskTitle: e.task.title,
    projectId: e.task.project.id,
    projectName: e.task.project.name,
    projectAlias: e.task.project.alias,
    workspaceId: e.task.project.workspace.id,
    workspaceName: e.task.project.workspace.name,
    worktreePath: e.worktreePath,
    startedAt: e.startedAt,
  }));
}
```

**Confidence:** HIGH — derived from existing Prisma patterns in codebase.

### Pattern 4: Terminal Reuse in MissionCard

**What:** `TaskTerminal` is already SSR-safe when loaded via `next/dynamic`. Each `MissionCard` wraps the dynamic import.

**Critical constraint (from Phase 26 decision):** `TaskTerminal` requires `next/dynamic({ ssr: false })` — xterm.js accesses `window` at import time.

**Adaptation note:** `TaskTerminal` requires a `worktreePath` prop to render the terminal (it renders a "no worktree" state when undefined). Since PTY executions in missions may have `worktreePath: null` (tasks without a baseBranch), the MissionCard must pass a non-null fallback — or `TaskTerminal` must be extended/wrapped to skip the worktreePath guard for mission use. The cleanest approach is to pass `worktreePath ?? task.project.localPath` from the server action (include `localPath` in the return shape).

**MissionTerminal dynamic import pattern:**

### Pattern 5: Polling for Status Changes

**What:** Client component polls `getActiveExecutionsAcrossWorkspaces` every 4 seconds. When an execution disappears from the result, it transitions its card to "completed/stopped" state, then removes it after 3 seconds.

```typescript
// In missions-client.tsx
useEffect(() => {
  const timer = setInterval(async () => {
    const fresh = await getActiveExecutionsAcrossWorkspaces();
    setCards((prev) => {
      // Cards in prev but not in fresh → transition to removing state
      // Cards in fresh but not in prev → add as new card
      // ...
    });
  }, 4000);
  return () => clearInterval(timer);
}, []);
```

**CRITICAL — timer cleanup:** Per `.claude/rules/process-lifecycle.md`: all `setInterval` must be cleared in `useEffect` return. The poll timer must be cleared on component unmount.

**Card fade-out pattern:** Use a `removingIds: Set<string>` state. When an execution leaves the RUNNING set, add its card to `removingIds`. After 3 seconds (`setTimeout`), remove from `cards`. The `setTimeout` must also be tracked and cleared on unmount.

### Anti-Patterns to Avoid

- **Forgetting `next/dynamic({ ssr: false })` on TaskTerminal:** xterm.js will crash SSR. Every consumer must wrap with dynamic import.
- **Using `<SelectValue />`:** Per ui.md, always use a manual `<span>` inside `<SelectTrigger>`.
- **Polling without cleanup:** `setInterval` in `useEffect` must return a cleanup that calls `clearInterval`. Same for the 3-second `setTimeout` used for fade-out.
- **Mutating the `worktreePath` issue:** `TaskTerminal` renders "no worktree" state when `worktreePath` is nullish. The server action must also include `project.localPath` as a fallback so every mission card gets a valid path to connect to the PTY session.
- **Not passing `id` to `DndContext`:** Per ui.md DnD Kit rule — always pass a stable `id` prop to prevent hydration mismatch.
- **Card order vs. polling conflict:** When polling replaces the card list with fresh data, it must preserve the user's drag-reordering. Merge logic: use the existing ordered array as the base, adding new arrivals at the end and removing gone executions.

## Common Pitfalls

### Pitfall 1: TaskTerminal worktreePath Guard
**What goes wrong:** `TaskTerminal` returns the "no worktree" placeholder when `worktreePath` is null/undefined. Tasks without a `baseBranch` (most tasks in this project) have `worktreePath: null` on their execution record.
**Why it happens:** `TaskTerminal` uses `if (!worktreePath) return <placeholder>` as its early return guard.
**How to avoid:** In `getActiveExecutionsAcrossWorkspaces`, include `task.project.localPath` in the return shape. Pass `execution.worktreePath ?? execution.projectLocalPath` to the terminal component. This gives the terminal a valid cwd to connect to even without a worktree.
**Warning signs:** All mission cards showing "无工作区" instead of a live terminal.

### Pitfall 2: Poll Merge Logic Losing Card Order
**What goes wrong:** Polling replaces the card list with `fresh` data from server, destroying the drag-reorder the user applied.
**Why it happens:** Naively setting `setCards(fresh)` ignores local ordering.
**How to avoid:** Use ordered merge:
```typescript
setCards(prev => {
  const freshIds = new Set(fresh.map(e => e.executionId));
  const prevIds = new Set(prev.map(c => c.executionId));
  // Keep existing ordered cards, remove gone ones (they go to removingIds)
  const retained = prev.filter(c => freshIds.has(c.executionId));
  // Append newly appeared executions
  const added = fresh.filter(e => !prevIds.has(e.executionId));
  return [...retained, ...added];
});
```
**Warning signs:** Card order resets every 4 seconds.

### Pitfall 3: Timer Leak on Unmount
**What goes wrong:** Navigation away from /missions leaves an active `setInterval` polling the server every 4 seconds. The 3-second `setTimeout` for fade-out also leaks if user navigates mid-fade.
**Why it happens:** React does not automatically clear timers.
**How to avoid:** Track both in `useRef` arrays; clear all in `useEffect` cleanup.
**Warning signs:** Console errors about updating unmounted component state.

### Pitfall 4: FitAddon Not Refitting After Grid Layout Change
**What goes wrong:** When user switches from 2×2 to 3×2, cards become smaller but terminals don't resize.
**Why it happens:** `TaskTerminal`'s `ResizeObserver` handles container resize, but it fires after the DOM has settled. CSS transitions may delay DOM measurement.
**How to avoid:** After changing layout preset, trigger a brief forced layout cycle. `TaskTerminal` already uses `ResizeObserver` — this should work automatically once the card's container dimensions change. No manual fit call needed if cards use CSS `h-full w-full`. Verify in testing.
**Warning signs:** Terminal content clipped or scrollbars appear inside card.

### Pitfall 5: dnd-kit `id` Prop Missing
**What goes wrong:** Hydration mismatch warning from dnd-kit when `DndContext` has no stable `id`.
**Why it happens:** dnd-kit generates a random id server-side that differs from client.
**How to avoid:** Always pass `id="missions-dnd"` to `<DndContext>`. Documented in ui.md.

### Pitfall 6: stopPtyExecution vs Card Removal Race
**What goes wrong:** User clicks stop → `stopPtyExecution` called → card immediately removed from local state → but polling also fires and tries to remove it again.
**Why it happens:** Both the stop handler and the poll detect the execution is gone.
**How to avoid:** When the stop button is clicked, immediately add the card to `removingIds` (don't remove from `cards` yet). The poll's merge logic will then find the executionId absent from fresh data and skip it (since it's already in `removingIds`). After 3 seconds, remove from both.

### Card fade-out with setTimeout cleanup
```typescript
// In missions-client.tsx
const [removingIds, setRemovingIds] = useState<Set<string>>(new Set());
const fadeTimers = useRef<Map<string, ReturnType<typeof setTimeout>>>(new Map());

const startFadeOut = useCallback((executionId: string) => {
  if (removingIds.has(executionId)) return; // already fading
  setRemovingIds(prev => new Set([...prev, executionId]));
  const t = setTimeout(() => {
    setCards(prev => prev.filter(c => c.executionId !== executionId));
    setRemovingIds(prev => { const n = new Set(prev); n.delete(executionId); return n; });
    fadeTimers.current.delete(executionId);
  }, 3000);
  fadeTimers.current.set(executionId, t);
}, [removingIds]);

// Cleanup on unmount
useEffect(() => {
  return () => {
    fadeTimers.current.forEach(t => clearTimeout(t));
  };
}, []);
```

## Open Questions

1. **TaskTerminal worktreePath fallback for tasks without baseBranch**
   - What we know: `worktreePath` is null for tasks without `baseBranch`. `TaskTerminal` guards on this with an early return.
   - What's unclear: Whether passing `project.localPath` as the fallback `worktreePath` is semantically correct (it IS the cwd for the PTY session).
   - Recommendation: Include `projectLocalPath` in `ActiveExecutionInfo`; pass `worktreePath ?? projectLocalPath` as the `worktreePath` prop to `TaskTerminal`. This is safe because PTY sessions always have a valid cwd regardless of worktree.

2. **Sidebar placement for /missions link**
   - What we know: The sidebar has workspace-scoped footer links (Notes, Assets, Archive). The missions link is global.
   - What's unclear: Whether to place it above the workspace list, in the footer always-visible area, or as a separate nav section.
   - Recommendation: Add it to the footer section unconditionally (before the workspace-scoped conditional block). This mirrors how Settings is handled in the collapsed view — always visible.
