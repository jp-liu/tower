# Milestone v0.3: 全局搜索增强

**Status:** SHIPPED 2026-03-30
**Phases:** 8-10
**Total Plans:** 4

## Overview

Extended the search system from fixed-type (task/project/repository) to full cross-type search covering notes and assets. Added asset description metadata, server-side search logic with FTS5 full-text search and graceful fallback, MCP tool parity, and a six-tab search dialog with grouped "All" rendering and content snippets.

## Phases

### Phase 8: Asset Description Schema
**Goal**: Assets carry a description that users can populate at upload time and that search can match against
**Depends on**: Nothing (schema-first phase)
**Requirements**: ASSET-01, ASSET-02
**Plans**: 1 plan

Plans:
- [x] 08-01: Schema + data layer + UI for asset description field

**Success Criteria**:
  1. User can type a description in the upload dialog before submitting an asset
  2. The submitted description is persisted to the database and visible after page reload
  3. Pre-existing assets without a description are not broken
  4. The upload dialog form rejects submission when the description field is missing

### Phase 9: Search Actions Expansion
**Goal**: Server-side search logic covers notes and assets across all projects, and MCP tool reflects the same capabilities
**Depends on**: Phase 8
**Requirements**: SRCH-01, SRCH-02, SRCH-03, SRCH-04
**Plans**: 1 plan

Plans:
- [x] 09-01: Note/asset/all search branches + MCP parity + test coverage

**Success Criteria**:
  1. globalSearch with category "note" returns notes via FTS5
  2. globalSearch with category "asset" returns assets by filename or description
  3. globalSearch with category "all" returns all five types grouped and capped
  4. MCP search tool accepts "note", "asset", and "all" categories
  5. Malformed FTS5 query falls back to LIKE search without crashing

### Phase 10: Search UI Extension
**Goal**: Users can run a single search and see all matching content across every type in one unified view, or narrow to a specific type via tabs
**Depends on**: Phase 9
**Requirements**: SUI-01, SUI-02, SUI-03, ASSET-03
**Plans**: 2 plans

Plans:
- [x] 10-01: Data layer — snippet field on SearchResult + i18n key additions
- [x] 10-02: UI — six-tab search dialog with grouped All rendering + snippets + component tests

**Success Criteria**:
  1. Search dialog shows six tabs: All, Task, Project, Repository, Note, Asset
  2. "All" tab displays results in named sections (one per type)
  3. Note search results display content snippet (~80 chars)
  4. Asset search results display description beneath filename
  5. All new UI elements support Chinese and English

## Milestone Summary

**Key Decisions:**
- ProjectAsset.description as nullable `String? @default("")` — avoids NOT NULL constraint on existing rows
- Inline raw SQL in search-actions.ts instead of extending fts.ts — fts.ts must stay Next.js-free
- Promise.allSettled for "all" mode fan-out — single SQLITE_BUSY must not drop all results
- search-actions.ts and search-tools.ts updated in same commit — they share no code and divergence is silent
- textarea + ReactMarkdown fallback (from v0.2) — React 19 compat concern validated

**Issues Resolved:**
- FTS5 malformed query crash — added try/catch with LIKE fallback
- Path traversal in uploadAsset — added path.basename sanitization, DB validation, and containment check
- deleteAsset orphaned files — now deletes physical file before DB record
- Search result navigation — notes and assets now navigate to their respective pages instead of the board

**Technical Debt Incurred:**
- Duplicate search logic between search-actions.ts and search-tools.ts (should extract to shared module)
- useEffect race condition in search dialog (stale results can overwrite newer ones)
- i18n.tsx approaching 800-line limit (should extract translation dicts to separate files)
