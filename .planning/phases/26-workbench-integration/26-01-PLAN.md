Implementation steps (per D-04, D-05, INT-01, INT-02):

4. Send-back: if task.status === "IN_REVIEW", update to IN_PROGRESS
5. Build full prompt string (inline the logic from stream/route.ts `buildExecutionPrompt`):

6. Prepare instructions file if task.promptId (or selectedPromptId) is set:
   - Look up AgentPrompt by selectedPromptId ?? task.promptId
   - Write to temp file: `mkdtemp` + `writeFile`
7. Create worktree if task.baseBranch set:
   - `createWorktree(task.project.localPath, taskId, task.baseBranch)` → { worktreePath, worktreeBranch }
8. Create TaskExecution row:
   ```typescript
   const execution = await db.taskExecution.create({
     data: {
       taskId,
       agent: "CLAUDE_CODE",
       status: "RUNNING",
       startedAt: new Date(),
       worktreePath: resolvedWorktreePath ?? null,
       worktreeBranch: resolvedWorktreeBranch ?? null,
     },
   });
   ```
9. Build Claude CLI args (per D-05, INT-02 — NO --output-format stream-json, NO --print -):
   ```typescript
   const claudeArgs = ["--dangerously-skip-permissions"];

   ```
10. Create PTY session via createSession:
    ```typescript
    createSession(
      taskId,
      "claude",
      claudeArgs,
      cwd,          // worktreePath if set, else localPath
      () => {},     // onData: ws-server handles broadcast; no-op here
      async (exitCode) => {
        // INT-03: status update on PTY exit
        await db.taskExecution.update({
          where: { id: execution.id },
          data: { status: exitCode === 0 ? "COMPLETED" : "FAILED", endedAt: new Date() },

        if (exitCode === 0) {
          await db.task.update({ where: { id: taskId }, data: { status: "IN_REVIEW" } });

          await rm(tempDir, { recursive: true, force: true }).catch(() => {});
        }
      }
    );
    ```

Note on onData: pass a no-op `() => {}` — the ws-server.ts already handles broadcasting to connected WebSocket clients from the PTY session's onData. The session-store creates the session and ws-server reads from it. Wait — re-read ws-server.ts carefully.

IMPORTANT: Looking at ws-server.ts line 102-116, when a new WS connection comes in and no session exists, it calls createSession with `makeBatchedSender(ws)` as onData. But startPtyExecution creates the session BEFORE any WS connection. This means the session is created with a no-op onData, and when the WS connects, it replays the buffer but new data goes nowhere.

Fix: ws-server.ts needs to handle the case where a session already exists (was pre-created by startPtyExecution). In handleConnection, if session already exists AND has a live onData that's a no-op, update its onData to the new WS's batchedSender. This requires making onData updatable on PtySession.

Actually, looking at pty-session.ts: the onData callback is captured in the constructor closure — it's not replaceable without modifying PtySession. The cleaner fix: in ws-server.ts handleConnection reconnect path, the session already exists — replay buffer works. But for new data after reconnect, we need the PTY to broadcast to the new ws.

Simplest correct fix: Add a `setDataListener(fn: (data: string) => void): void` method to PtySession, and in ws-server.ts handleConnection, call `session.setDataListener(makeBatchedSender(ws))` whether reconnecting or new connection. Then in startPtyExecution, pass a no-op for onData since ws-server will set the real listener on connect.

This requires touching pty-session.ts too. Update PtySession to support setDataListener:
```typescript
// Add to PtySession class:
private _onData: (data: string) => void;

// In constructor, store callback:
this._onData = onData;
this._pty.onData((data) => {
  this._buffer += data;
  ...
  this._onData(data);  // call stored, not closure
});

setDataListener(fn: (data: string) => void): void {
  this._onData = fn;
}
```

Then in ws-server.ts handleConnection (both reconnect and new connection paths), call:
```typescript
session.setDataListener(makeBatchedSender(ws));
```
And on ws.close, reset to no-op:
```typescript
session.setDataListener(() => {});
```

This means startPtyExecution can safely pass `() => {}` as onData — ws-server sets the real listener on connect.
