## REQ-01: Session Knowledge Extraction (Dreaming)

**Goal:** When a task execution ends, automatically extract reusable knowledge (pain points, solutions, patterns) and persist as project notes, with visibility in execution history and daily reports.

### Current State

- `captureExecutionSummary()` in `src/lib/execution-summary.ts` captures git log, stats, terminal buffer
- Phase 2 calls `generateSummaryFromLog()` which produces a one-line 50-char summary via `claude -p`
- Execution timeline shows: status, duration, git stats, expandable git log and terminal log
- `ProjectNote` model exists with `title`, `content`, `category`, `projectId`, `taskId`

### Requirements

#### REQ-01a: Enhanced AI Summary Generation

After the existing one-line summary is generated, fire a second (longer) AI analysis:

**Input:** Terminal log (last 10KB) + git log + git diff stat
**Output:** Structured JSON:
```json
{
  "summary": "string (1-2 sentences, current behavior)",
  "insights": [
    {
      "type": "pitfall" | "pattern" | "solution" | "tip",
      "title": "short title (< 30 chars)",
      "content": "detailed description (100-300 chars)"
    }
  ],
  "shouldCreateNote": true | false
}
```

**Rules:**
- Only create notes when `shouldCreateNote === true` (AI determines significance)
- Typical triggers: resolved tricky bugs, discovered important patterns, found reusable solutions
- Trivial changes (typo fix, config tweak) should NOT produce notes
- Timeout: 60s (longer than current 30s since deeper analysis)
- Failure is non-blocking (fire-and-forget, same as current summary)

#### REQ-01b: Auto-create ProjectNote from Insights

When `shouldCreateNote === true`:
- Create a `ProjectNote` with:
  - `title`: AI-generated (from insights)
  - `content`: Markdown combining all insights
  - `category`: `"session-insight"` (new category)
  - `projectId`: from task's project
  - `taskId`: the current task
- Store the note ID in a new field on `TaskExecution`: `insightNoteId`

#### REQ-01c: Execution Timeline UI Enhancement

Below the existing summary line in each execution card, add:

```
摘要: 修复了 WebSocket 连接超时问题
归纳: [WebSocket 超时处理方案]  ← clickable, opens note preview
```

- If `insightNoteId` exists, show the "归纳" row with the note title
- Click behavior: inline expand showing note content (markdown rendered), with a "查看完整笔记" link to navigate to notes tab
- If no insight note, don't show the row

#### REQ-01d: Daily Summary Enhancement

Extend `daily_summary` output format with a new section:

```json
{
  ...existing fields,
  "insights": [
    {
      "workspaceName": "string",
      "projectName": "string",
      "taskTitle": "string",
      "noteTitle": "string",
      "noteId": "string",
      "category": "session-insight"
    }
  ]
}
```

Query: All `ProjectNote` with `category = "session-insight"` created within the date range, joined with task/project/workspace info.

## REQ-02: Project Import & Migration

**Goal:** Streamline project onboarding with a dedicated import flow and optional path migration for scattered projects.

### Current State

- "New Project" dialog in top-bar supports: name, alias, description, gitUrl, localPath, browse button
- `resolveGitLocalPath()` maps git URL to local path via configured rules (Phase 12)
- FolderBrowserDialog browses filesystem and detects git repos

### Requirements

#### REQ-02a: Separate Import from Create

Split current "New Project" into two entry points:

1. **Create Project (新建项目)**
   - Input: git URL (required)
   - Auto-resolve: local path via git rules (editable)
   - If path doesn't exist: clone
   - If path exists: use as-is
   - No browse button (path is derived from git URL)

2. **Import Project (导入项目)**
   - Entry: browse button opens FolderBrowserDialog
   - Or: same as clicking an unregistered project in sidebar browse
   - Auto-detect git remote from selected path
   - Auto-fill name from folder name or git repo name
   - User can edit all fields before confirming

#### REQ-02b: Migration Feature (on Import only)

When importing a project, show an optional toggle:

```
☐ 迁移项目到规范路径
  当前路径: /Users/foo/Downloads/my-project
  迁移后:   /Users/foo/project/github.com/org/my-project  ← editable
```

**UI 行为:**
- Toggle OFF by default
- When ON: show "迁移后路径" field (derived from git URL via `resolveGitLocalPath()`)
- User can edit the migration target path
- If no git remote detected: migration toggle hidden (can't derive target path)
- Show warning text: "迁移用时随项目大小而定，大型项目可能需要较长时间"

**迁移执行细则:**

1. **前置检查 (开启 toggle 时立即验证，不满足则阻止确认):**
   - 目标路径不能已存在（存在则报错 "目标路径已存在，请修改"）
   - 目标路径的父目录必须可写（否则报权限错误）
   - 源路径必须存在且可读
   - 该项目下不能有 RUNNING 状态的 execution — 提示："请先停止所有运行中的任务后再迁移"
   - 该项目下不能有活跃的 PTY session — 提示："请先关闭所有终端会话后再迁移"
   - 如果项目存在 worktree（`.worktrees/` 目录有内容）— 提示："请先清理所有 worktree 后再迁移"
   - **原则：不自动停止/清理任何东西，只告诉用户哪些阻塞了迁移，让用户自己处理好再来**

2. **执行步骤 (confirm 后按顺序):**
   ```
   Step 1: 再次验证前置条件（防止检查后到点击确认之间状态变化）
   Step 2: 创建目标路径的父目录 (mkdir -p)
   Step 3: 执行 fs.rename(source, target) — 同文件系统下是原子操作，瞬间完成
   Step 4: 验证目标路径存在且包含 .git
   Step 5: 创建项目记录 (localPath = 新路径)
   ```

3. **为什么用 fs.rename 而不是 cp:**
   - `fs.rename()` 在同文件系统上 = POSIX `rename()` 系统调用
   - 它是**原子操作**：只修改目录入口指针，不移动实际文件数据
   - 无论项目多大（node_modules 10万个文件）都是瞬间完成
   - 要么完全成功，要么完全失败，不存在"中间状态"
   - 不需要 cp + rm 方案（那才是慢且危险的）
   - **限制：源和目标必须在同一文件系统/磁盘分区** — 本产品只操作本机本地项目，这个条件天然满足
   - 如果 `fs.rename` 报 EXDEV 错误（理论上不该出现）：直接报错 "源路径与目标路径不在同一磁盘，无法迁移"，不尝试 cp fallback

4. **错误处理:**
   - `fs.rename` 失败: 原子操作，失败 = 源路径完好无损，无需回滚
   - `mkdir -p` 失败: 报错 "无法创建目标目录，请检查权限"
   - Step 4 验证失败（极端情况）: 报错并给出两个路径让用户手动检查
   - **核心安全保证：rename 的原子性意味着不会出现"源被删了目标没建好"的情况**

5. **进度反馈 (UI):**
   - 点击确认后，按钮变为 loading 状态 + 文字 "正在迁移..."
   - 实际上 rename 是瞬间完成的（< 1ms），loading 主要覆盖 mkdir + DB 写入的网络延迟
   - 成功: toast.success("项目已迁移到新路径")
   - 失败: toast.error(具体错误信息)

6. **迁移后状态:**
   - 项目 `localPath` 指向新路径
   - Git 配置无需修改（`.git` 目录整体随项目移动，内部用相对路径）
   - 原路径不保留任何残留（symlink 也不创建，避免幽灵引用）
   - 前置条件已确保无 worktree/session，所以不需要迁后清理

7. **边界情况:**
   - 项目已在规范路径: 目标路径 = 当前路径时，禁用确认按钮 + 提示 "已在规范路径，无需迁移"
   - 导入时用户手动改了迁移目标路径为不存在的深层目录: mkdir -p 自动创建
   - 不勾选迁移: 正常导入项目，localPath = 浏览选择的原始路径，不做任何文件移动
   - 取消整个导入弹窗: 不创建项目、不移动文件，什么都不发生

## REQ-03: Asset Preview & File Operations

**Goal:** Support inline preview for images and text files; provide "reveal in Finder" for all files.

### Current State

- `AssetItem` shows thumbnail for images, icon for others
- Actions: download + delete
- Supported MIME types: png, jpg, gif, webp, svg, pdf, txt, md, json

### Requirements

#### REQ-03a: Image Lightbox Preview

- Click image thumbnail → open fullscreen lightbox overlay
- Controls: close (X or Escape), zoom in/out, drag to pan
- Support keyboard: Escape to close, arrow keys if multiple images
- Use a lightweight library (e.g., `yet-another-react-lightbox`) or custom dialog

#### REQ-03b: Text/Markdown Preview

- For `.txt`, `.md`, `.json` files: click → open preview dialog
- `.md` files render as Markdown (reuse existing markdown renderer from assistant chat)
- `.txt` and `.json` show as monospace preformatted text
- Dialog: modal with title (filename), scrollable content, close button

#### REQ-03c: Reveal in Finder

- New action button: "在文件夹中显示" (folder icon)
- Implementation: POST to a new internal API that runs `open -R <filepath>` (macOS) / `xdg-open` (Linux)
- Only show this button when running locally (always true for Tower)

#### REQ-03d: Action Button Layout

Replace current download + delete with:

```
[预览] [在文件夹中显示] [删除]
```

- Preview: eye icon — opens lightbox (image) or text preview dialog (text)
- Reveal: folder icon — opens containing folder in system file manager
- Delete: trash icon (existing, keep red hover)
- Download removed (users can use "reveal in finder" then copy)

## REQ-05: Session Auto-Upload Hook (Live Asset Capture)

**Goal:** Claude Code 执行过程中产生的文件资源（图片、文档等）自动上传为任务资源。

### 机制

```
Tower PTY 启动 Claude Code
  ↓ 注入: TOWER_TASK_ID, TOWER_API_URL
  ↓
Claude Code PostToolUse Hook 触发
  ↓ 全局 hook 脚本检测 TOWER_TASK_ID 是否存在
  ↓ 不存在 → exit 0（非 Tower 会话，不干预）
  ↓ 存在 → 检测本次工具操作是否产出/引用了目标文件类型
  ↓ 匹配 → curl POST 到 TOWER_API_URL 上传文件，关联 taskId
  ↓
文件出现在任务资源列表
```

### Requirements

#### REQ-05a: Hook 脚本 & 全局注册

**Hook 脚本：** `~/.claude/hooks/tower-auto-upload.sh`

```bash
#!/bin/bash
# 非 Tower 会话直接退出
[ -z "$TOWER_TASK_ID" ] && exit 0

# 从 stdin 读取 PostToolUse event JSON
# 解析 tool_name, file_path, etc.
# 检查文件扩展名是否在允许列表中
# 如果匹配，调用 TOWER_API_URL 上传
```

**注册方式：**
- Tower 设置页提供 "安装 Hook" 按钮（一键写入全局 settings.json）
- 或 Tower 首次启动时检测，未安装则提示引导

**注册内容（写入 `~/.claude/settings.json`）：**
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Bash",
        "command": "~/.claude/hooks/tower-auto-upload.sh"
      }
    ]
  }
}
```

#### REQ-05b: 文件类型配置

新增 SystemConfig key: `hooks.autoUploadTypes`

**默认值：**
```
png,jpg,jpeg,gif,webp,svg,md,txt,pdf,docx,xlsx,csv
```

- 用户可在设置页修改（逗号分隔扩展名列表）
- Hook 脚本启动时从 Tower API 拉取当前配置（或通过额外环境变量 `TOWER_UPLOAD_TYPES` 注入）

#### REQ-05c: 上传 API

新增内部 API: `POST /api/internal/assets/upload-from-hook`

**Request:**
- `taskId` (from env)
- `filePath` (absolute path on disk)
- 或 `file` (multipart, hook 直接上传文件内容)

**Behavior:**
- 验证 taskId 有效
- 验证文件扩展名在允许列表
- 复制文件到 `data/assets/{projectId}/`（通过 task 找到 projectId）
- 创建 Asset 记录，关联 taskId
- 返回 assetId

#### REQ-05d: 环境变量重命名

全局重命名（breaking change，开发阶段无需兼容）：
- `AI_MANAGER_TASK_ID` → `TOWER_TASK_ID`
- `AI_MANAGER_API_URL` → `TOWER_API_URL`（如不存在则新增）
- 同步修改所有引用：PTY spawn、signal file、internal API routes

## REQ-06: 项目资源展示任务资源 & 任务概览抽屉

interface TaskOverviewDrawerProps {
